## Project description

This web platform serves as a central hub where users can present, discover, and further develop their innovative projects.

Users can upload their own projects and present them in detail, including descriptions, files, and additional information. At the same time, they can browse through other users' projects and find inspiration.

A central component of the platform is interaction within the community: projects can be commented on, discussed, and rated. Users can purchase and download projects or associated files to reuse them or use them as a basis for their own developments. Overall, the platform combines presentation, collaboration, and monetization.

## Inspiration

> Many people have a notebook full of project ideas вҖ” a soil moisture meter, a RAG chatbot, a mechanical keyboard вҖ” but no central place where they can record them, present them, or figure out what it would cost to bring several of these ideas to life at once in terms of parts, services, and expertise.
>
> **VentureCanvas solves exactly this problem:** a small community gallery where every user can publish their own projects and collect interesting ideas from others. The "Collection" view does the heavy lifting: it summarizes the required skills, tools, APIs, and hardware for all saved projects, so the curator can see the entire shopping list at a glance.

## 3 В· Architecture

VentureCanvas follows a strict three-layer architecture with a thin UI on top. Arrows point downward вҖ” no upward dependencies.

```mermaid
flowchart TB
    UI["<b>ui/</b>  <i>View + Controller</i><br/>pages.py вҖ” NiceGUI @ui.page bindings<br/>controllers.py вҖ” Auth/Home/Project/Collection<br/>session_state вҖ” browser login state"]
    SVC["<b>services/</b>  <i>business logic, class-based</i><br/>auth_service.py В· project_service.py<br/>collection_service.py В· errors.py"]
    DAO["<b>data_access/</b>  <i>Repository / DAO + Facade</i><br/>db.py (Database facade) В· dao.py (BaseDAO + 3)<br/>password_hasher.py В· seed.py (ProjectSeeder)"]
    DOM["<b>domain/</b>  <i>Model вҖ” SQLModel entities</i><br/>models.py вҖ” User В· Project В· CollectionItem"]

    UI  -- "plain Python calls"       --> SVC
    SVC -- "DAOs + session_scope"     --> DAO
    DAO -- "SQLModel / SQLAlchemy 2"  --> DOM

    classDef layer fill:#fafafa,stroke:#111,stroke-width:1.5px,color:#111,rx:6,ry:6;
    class UI,SVC,DAO,DOM layer;
```

### Named patterns (rubric ticks)

| Pattern | Where |
|---|---|
| **MVC** | `domain/` (Model) В· `ui/pages.py` (View) В· `ui/controllers.py` (Controller) |
| **Repository / DAO** | `data_access/dao.py` вҖ” `BaseDAO` + `UserDAO` / `ProjectDAO` / `CollectionDAO` |
| **Facade** | `data_access/db.py::Database` вҖ” hides the SQLModel engine behind `session_scope` + `init_schema_and_seed` |
| **Composition Root** | `application.py::VentureCanvasApplication` вҖ” builds the whole object graph from one constructor |

## 4 В· Database & ORM

Three tables mapped to three SQLModel classes; SQLite in development, PostgreSQL in production (set `DATABASE_URL`).

```mermaid
erDiagram
    USERS ||--o{ PROJECTS         : "owns"
    USERS ||--o{ COLLECTION_ITEMS : "saves"
    PROJECTS ||--o{ COLLECTION_ITEMS : "saved as"

    USERS {
        int       id              PK
        string    username
        string    email           UK
        string    password_hash
        datetime  created_at
    }
    PROJECTS {
        int       id              PK
        int       owner_id        FK
        string    title
        string    description
        Category  category
        string    required_skills
        string    required_tools
        string    required_apis
        string    required_hardware
        datetime  created_at
        datetime  updated_at
    }
    COLLECTION_ITEMS {
        int       id              PK
        int       user_id         FK
        int       project_id      FK
        datetime  added_at
    }
```

<sub><i>Relationship in shorthand:&nbsp; <code>users в”Җ1:Nв”Җ> projects в”Җ1:Nв”Җ> collection_items <в”Җ1:1в”Җ users</code> &nbsp;(unique <code>user_id, project_id</code>)</i></sub>

### ER diagram

<div align="center">
  <img src="docs/architecture-diagrams/er_diagram.png" alt="ER diagram" width="500" />
</div>

Every column carries an explicit `Field(...)` declaration (`min_length`, `max_length`, `unique`, `index`, `foreign_key`): this is the single declarative source of truth and it shapes the generated SQL schema. Because SQLModel's `table=True` models skip pydantic validation at construction, the length/format rules are enforced in the **service layer**, not on the model вҖ” `AuthService` validates the username, email and password; `ProjectService` validates the title and description, bounds the four requirement lists, and coerces the category to the closed `Category` set. The database enforces the structural guarantees: `unique` on username/email and the `(user_id, project_id)` `UniqueConstraint` on the collection.

## 8 В· How to run

**Requires Python 3.11+.**

```bash
# 1. Clone
git clone https://github.com/atanasovmi/VentureCanvas.git
cd VentureCanvas

# 2. (Optional) copy the env template
cp .env.example .env        # defaults to SQLite at data/venturecanvas.db

# 3. Create a virtualenv and install
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt

# 4. Run
python -m venturecanvas      # opens http://localhost:8080
```

> [!NOTE]
> The first launch automatically creates the SQLite file and seeds one demo user plus sample projects across the five categories.

> [!TIP]

## 12 В· Design decisions

- рҹ”Ғ &nbsp;**Session-injection DAOs.** Services open a `Database.session_scope` and pass the session to DAOs, so one business operation equals one transaction.
- рҹ”җ &nbsp;**PasswordHasher in `data_access/`.** A hash is the at-rest representation of a secret вҖ” a persistence concern. Placing it there lets both `AuthService` and `ProjectSeeder` depend on it without crossing the layer boundary upward.
- рҹҺҜ &nbsp;**Enum `Category`.** Replaces the old repo's free-text category column. Rejects unknown values at the service boundary and makes the filter chips a closed set.
- вҸі &nbsp;**`expire_on_commit=False` on sessions.** Services return entities whose scalar attributes are still readable after the session closes, which keeps controllers and pages session-agnostic.
- вңӮпёҸ &nbsp;**Twelve features, full stop.** Comments, free-text search, pagination, profile edit, admin panel and animated backgrounds from the previous stack were cut. Every feature we kept has a page, an owner, and a test.
